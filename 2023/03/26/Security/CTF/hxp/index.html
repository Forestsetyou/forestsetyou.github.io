测试

在生产环境下测试：

548行的 app.render 函数中有语句：

该语句在生产环境下会返回”True”，因此促成了后续模板渲染结果被缓存，二次解析被忽略。

“ejs.js”第210行的 handleCache 函数将获取返回的内容，并实现相应模板的缓存。

exports.compile 赋值了模板类的 compile 函数，该函数完成了整个模板的解析过程，其解析思路是根据类中提供的opts设置先完成对模板内容的解析，然后根据解析的结果拼接出固定的函数，该函数将依据本次解析的内容构造出模板渲染的结果，并且该函数将随之被存入与当前模板名对应的缓存中。之后再次渲染该模板时将不再重新解析，而是直接从缓存中拿出构造函数，生成模板的渲染内容，并返回。

其中的参数如”name”是被单独识别并替换的，而除此之外的其他部分包括 openDelimiter, delimiter 等参数都被缓存了起来，因此改模板替换name的效果是有效的，但其他参数的篡改是做不到的。

ejs内部是根据传入的渲染设置，动态拼接出一个字符串形式的JS函数，然后将该字符串传入 Function 的构造函数来获取一个崭新的函数，该函数对应字符串。当断点运行到当前环境时，可以用 Console.log(fn) 来展示函数的具体内容，DEV工具中鼠标悬浮查看函数体是找不到任何内容的，在控制台中也不能直接用 fn 查看函数体。

随后回到 handleCache 函数中，fn最后被赋给了func，由于 options.cache 为True，模板的构造函数被缓存。最后模板的渲染结果被返回。

可以看出ejs为了生产环境的效率，会存储初次被传入的模板解析结果，并省略后续的解析。为了更换模板分隔符，需要在首次传入就传入参数 delimiter=? ，但由于源码”app.js”中：

最后的自动重定向会自动污染参数，因此需要用burp抓包来阻止该重定向。由于要生成模板就会调用该函数，因此重定向是无法用源码的信息绕过的，同时生成的模板名称是随机的，若不借助重定向的请求我们将无法获取新生成模板的信息，因此最后唯一的解法就是burp抓包阻止重定向，并发出一个有正确payload的包来初始化ejs的模板参数。

被缓存的函数的键是对应文件的路径，但这不代表缓存的函数被存在这些文件中，实际上它们被放在node服务器的缓存里。
